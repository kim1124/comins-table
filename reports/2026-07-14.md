## 17:00 KST - Tree, Row Expand, and Row Grouping compatibility research

### Sources Checked

- Repository source: `src/core.ts`, `src/index.tsx`, `styles.css`, `src/component-renderer.tsx`
- Repository tests and examples: flat row movement, selection, clipboard, pagination, lazy loading, infinite scrolling, and virtualization coverage
- Approved Tree Grid design: `docs/superpowers/specs/2026-07-14-summary-row-and-tree-grid-design.md`
- No external documentation or web source was required because the behavior under review is determined by this repository's current implementation.

### Confirmed Findings

- Tree Node expansion controls which nested data nodes are included in the visible pre-order row list; it changes hierarchy-aware sorting and prevents Tree V1 row movement, pagination, lazy loading, and infinite scrolling.
- Current row operations use a flat `state.rows` and fixed `rowHeight` virtual-window calculation. Flat row movement moves one data row, and clipboard/range selection indexes flat visible rows.
- Row Expand is therefore a separate detail-rendering feature: its detail must follow the owner row while normal flat sorting, pagination, infinite scrolling, lazy loading, and row movement continue to target only the owner row.
- Dynamic Row Expand content cannot reuse the Tree V1 fixed-height virtual calculation. It will require a separate variable-height virtual-layout design when that feature is scheduled.
- Row Grouping is a derived projection of flat rows and should retain separate group ids and expansion state rather than reuse `CominsTreeNode`.

### Decision Applied To Tree Scope

- Tree Grid implementation may proceed without a functional conflict, provided it keeps Tree node expansion, flat Row Expand state, and future grouping state separate.
- The Tree design now documents these boundaries and limits shared implementation to an accessible disclosure-button primitive.

### Residual Risks

- Row Expand and Row Grouping APIs are not designed or implemented in the Tree delivery; their future variable-height virtualization and group-operation semantics require dedicated design work.
- The existing Playwright-generated screenshots under `reports/artifacts/visual-typography/` remain untracked and preserved.
